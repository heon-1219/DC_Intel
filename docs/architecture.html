A beginner-friendly platform that predicts short-term stock direction (up / down / neutral) with calibrated confidence, plain-language reasoning, and an honest public win-loss record.

1System architecture

End-to-end pipeline: external data sources are polled by in-process APScheduler jobs, normalized into Redis (hot cache) and SQLite (system of record), assembled into feature vectors, scored by six per-timeframe models, and served by FastAPI to the React SPA.

Hosting (owner decision: free + local-first). v1 runs the docker-compose stack on localhost at $0, with the full sentiment model running locally. No paid cloud service is used. An optional free-cloud demo path exists (GCP Always-Free e2-micro, US regions only — the Seoul region asia-northeast3 is not free; or Railway free credit), which would drop to the smaller MiniLM sentiment model to fit 1 GB RAM — a deliberate feature-cut, not the v1 target. Yahoo Finance via yfinance is unofficial and can break without notice; the price layer is isolated behind a provider interface so a free fallback (pykrx / Finnhub) can be swapped in without touching downstream code.

2Signal data flow & explainability path

How each of the three signal families — technical, sentiment/intel, and economic calendar — becomes model input, and how every prediction’s reasoning travels back out to the UI as up to three plain-language evidence bullets whose contributions sum to 100%.

3User journey

The core loop in five screens: sign in, scan the dashboard, find a stock across exchanges, read a prediction with its reasoning, then check the model’s honest track record before trusting it.

4Update cadences

One hour of wall-clock time, one row per job. Dots are refresh events drawn at the midpoint of each configured range; the label on the left states the authoritative range. The bottom band shows the two triggers that are not on the 60-minute clock.

5Component interaction — real-time vs background

Two disjoint traffic classes share the same storage. Solid blue lines are user-triggered synchronous request paths (numbered in order); dashed violet lines are cadence-driven background work. The only meeting points are Redis and SQLite — background jobs never block a user request.

Scale-up seams (documented, not built in v1). APScheduler → Celery when jobs need their own processes; SQLite → PostgreSQL when concurrent writes contend (schema is kept compatible); polling → WebSocket in v2 for push updates. Each seam is isolated behind one module boundary so the swap is local.
